# DoTs Deep-Dive: Repo Architecture, OpenAPI Pipeline, and PR Walkthrough

---

## Part 1 — What Is DoTs?

DoTs (`@digitalocean/dots`) is the official DigitalOcean TypeScript SDK. It is an **auto-generated** REST client built with [Microsoft Kiota](https://github.com/microsoft/kiota) from DigitalOcean's [OpenAPI 3.0 specification](https://github.com/digitalocean/openapi).

The package gives you a fully typed, fluent API to talk to every DigitalOcean endpoint:

```typescript
const account = await client.v2.account.get();
const droplets = await client.v2.droplets.get();
```

## Part 3 — How OpenAPI Feeds Into DoTs

### The pipeline at a glance

```
┌──────────────────────────┐
│  digitalocean/openapi    │  Upstream: OpenAPI 3.0 YAML spec
│  (GitHub repo)           │  PRs add/change API endpoints
└──────────┬───────────────┘
           │  merged PR → CI publishes versioned YAML to
           │  DigitalOcean Spaces (CDN)
           ▼
┌──────────────────────────┐
│  DigitalOcean-public-    │  Artifact: DigitalOcean-public-{sha}.v2.yaml
│  {sha}.v2.yaml           │  (or latest: DigitalOcean-public.v2.yaml)
└──────────┬───────────────┘
           │  Downloaded by `make download-spec` or CI workflow
           ▼
┌──────────────────────────┐
│  kiota generate          │  Kiota CLI reads the YAML spec
│  -l typescript           │  Generates TypeScript client code
│  -d spec.yaml            │  Output: src/dots/
│  -c DigitalOceanClient   │
│  -o ./src/dots           │
└──────────┬───────────────┘
           │  Produces:
           │  • digitalOceanClient.ts  (client factory + interface)
           │  • models/index.ts        (all API types)
           │  • v2/**                  (request builders per path)
           │  • kiota-lock.json        (spec hash + Kiota version)
           ▼
┌──────────────────────────┐
│  npm run build (tsc)     │  Compiles .ts → .js alongside source
└──────────┬───────────────┘
           │
           ▼
┌──────────────────────────┐
│  @digitalocean/dots      │  Published npm package
│  on npm                  │
└──────────────────────────┘
```

### Step-by-step: what happens in CI

The GitHub Actions workflow `.github/workflows/typescript-client-gen.yml` is triggered manually with an `openapi_short_sha` input:

1. **Checkout** the dots repo.
2. **Changelist** — runs `scripts/openapi_changelist.sh` to diff merged PRs on `digitalocean/openapi` between the current SHA (from `DO_OPENAPI_COMMIT_SHA.txt`) and the target SHA. This goes into release notes.
3. **Clean** — `make clean` removes `digitalOceanClient.ts`, `models/`, `v2/`, `kiota-lock.json`.
4. **Download spec** — fetches `DigitalOcean-public-{sha}.v2.yaml` from DigitalOcean Spaces.
5. **Update SHA** — writes the new SHA to `DO_OPENAPI_COMMIT_SHA.txt`.
6. **Install Kiota** — `microsoft/setup-kiota@v0.5.0`.
7. **Generate** — `make generate` → `kiota generate -l typescript -d spec.yaml -c DigitalOceanClient -o ./src/dots`.
8. **Generate docs** — `npm run docs` (TypeDoc).
9. **Commit + PR** — pushes to branch `openapi-{sha}/clientgen` and opens a PR.

### Step-by-step: what you do locally

```bash
# 1. Remove old generated code
make clean

# 2. Download the latest spec from DO Spaces
make download-spec
# → creates ./DigitalOcean-public.v2.yaml

# 3. Generate client from spec
make generate
# → kiota generate -l typescript -d ./DigitalOcean-public.v2.yaml -c DigitalOceanClient -o ./src/dots

# 4. Compile TypeScript
npm run build

# 5. Run mocked tests to verify
make test-mocked
```

Or, to use a specific/local spec file:

```bash
SPEC_FILE=./my-custom-spec.yaml make generate
```

### The Kiota lock file

`src/dots/kiota-lock.json` records:
- **kiotaVersion**: `1.30.0` — the exact Kiota CLI version used
- **descriptionHash**: SHA-512 of the spec file that produced this generation
- **descriptionLocation**: `../../DigitalOcean-public.v2.yaml` (relative from `src/dots/`)
- **clientClassName**: `DigitalOceanClient`
- **language**: `TypeScript`
- **serializers/deserializers**: JSON, Text, Form, Multipart

This file lets you verify what spec + tool version produced the current code.

### What Kiota actually generates

Kiota reads every path and schema in the OpenAPI YAML and produces:

- **`digitalOceanClient.ts`** — A factory function `createDigitalOceanClient(requestAdapter)` that registers serializers/deserializers and returns a proxied client with a `.v2` property.
- **`v2/`** — A tree of request builder classes, one per API path. For example, `v2/droplets/` contains builders for `GET /v2/droplets`, `POST /v2/droplets`, `GET /v2/droplets/{droplet_id}`, etc. Each builder has methods like `.get()`, `.post()`, `.delete()` that construct and send the appropriate HTTP request.
- **`models/index.ts`** — TypeScript interfaces for every API model (Droplet, Database, App, Account, etc.) plus `create*FromDiscriminatorValue`, `serialize*`, and `deserializeInto*` functions for each. This is ~29,500 lines.

Kiota also does some automatic transformations:
- **snake_case → camelCase**: API parameters like `droplet_limit` become `dropletLimit` in the generated types.
- **Reserved word escaping**: `default` becomes `defaultEscaped`.

### The OpenAPI spec repo

The spec lives at [github.com/digitalocean/openapi](https://github.com/digitalocean/openapi). Teams at DigitalOcean submit PRs to add or modify endpoint definitions. When a PR merges, CI publishes the updated YAML to DigitalOcean Spaces, and the `dots` CI workflow can be triggered with the new SHA to regenerate the client.

## Part 4 — PR Walkthrough: Streaming + Multi-Host Services

This PR adds two hand-written modules on top of the Kiota-generated client. Neither is auto-generated — they are layered on top to solve problems that Kiota's generated code does not handle natively.

### Problem 1: Streaming responses

DigitalOcean's AI/inference endpoints (Gen AI, Serverless Inference, Agent Inference) return streaming responses — Server-Sent Events (SSE), newline-delimited JSON (NDJSON), or plain text streams. Kiota generates request builders that expect complete JSON responses; it has no built-in concept of consuming a chunked stream.

### Solution: `StreamingRequestAdapter`

The new `src/dots/streaming/` module provides:

**`StreamingRequestAdapter`** — wraps a Kiota `RequestAdapter` and provides a `stream()` method that:

1. Takes a `RequestInformation` (Kiota's request abstraction containing URL, method, headers).
2. Calls the browser/Node `fetch()` API directly (bypassing Kiota's send pipeline).
3. Gets the `ReadableStream` from the response body.
4. Reads chunks with a `TextDecoder`, buffering partial data.
5. Dispatches parsed payloads to `onData` / `onError` / `onComplete` callbacks.

The adapter supports three stream formats via the `StreamingResponseType` enum:

| Format | Content-Type | Parsing strategy |
|---|---|---|
| `TEXT_EVENT_STREAM` | `text/event-stream` | Split on `\n\n`, extract `data:` lines, JSON.parse each |
| `APPLICATION_NDJSON` | `application/x-ndjson` | Split on `\n`, JSON.parse each line |
| `TEXT_PLAIN` | `text/plain` | Split on `\n`, emit each line (optionally JSON.parse) |

**How the buffer works:**

```
Incoming chunks:     "data: {\"tok\":\"he"  +  "llo\"}\n\ndata: {\"tok\":\"world\"}\n\n"
                     ────────────────────     ─────────────────────────────────────────
After concatenation: "data: {\"tok\":\"hello\"}\n\ndata: {\"tok\":\"world\"}\n\n"
Split on \n\n:       ["data: {\"tok\":\"hello\"}", "data: {\"tok\":\"world\"}", ""]
                      ↑ emit as JSON object          ↑ emit as JSON object        ↑ keep as buffer
```

The key insight: the adapter always keeps the **last element** of the split as the buffer (it might be an incomplete chunk), and processes everything before it.

**Utility helpers in `streamingUtils.ts`:**

| Function | Purpose |
|---|---|
| `collectStream(adapter, requestInfo, options)` | Runs the stream to completion, collects all chunks into an array, returns `T[]` |
| `createStreamingAdapter(underlyingAdapter)` | Convenience factory: `new StreamingRequestAdapter(underlyingAdapter)` |
| `streamWithHandler(adapter, requestInfo, handler, options)` | Processes each chunk with a custom async handler, returns a Promise |

**Usage pattern:**

```typescript
import { FetchRequestAdapter } from "@microsoft/kiota-http-fetchlibrary";
import { createStreamingAdapter, collectStream, StreamingResponseType } from "@digitalocean/dots";

const adapter = new FetchRequestAdapter(authProvider);
const streamingAdapter = createStreamingAdapter(adapter);

const requestInfo = new RequestInformation();
requestInfo.URL = "https://inference.do-ai.run/v1/chat/completions";
requestInfo.httpMethod = HttpMethod.POST;

const chunks = await collectStream(streamingAdapter, requestInfo, {
    streamType: StreamingResponseType.TEXT_EVENT_STREAM,
});
```

### Problem 2: Multiple API hosts

The Kiota-generated client targets a single base URL: `https://api.digitalocean.com`. But DigitalOcean now has multiple services with different hosts:

| Service | Base URL | Auth |
|---|---|---|
| DigitalOcean API v2 | `https://api.digitalocean.com/v2` | API key (Bearer token) |
| Serverless Inference | `https://inference.do-ai.run` | Bearer token |
| Agent Inference | `https://{agent}.example.com/api/v1` | Optional Bearer token + custom headers |

### Solution: `ServiceManager` and factory functions

The new `src/dots/services/` module provides:

**Three factory functions**, each returning a `ServiceClient` (adapter + auth provider + config):

```typescript
// DigitalOcean API — uses DigitalOceanApiKeyAuthenticationProvider
const doClient = createDigitalOceanClient("do-api-key");

// Serverless Inference — uses BearerTokenAuthenticationProvider
const inferenceClient = createServerlessInferenceClient("inference-token");

// Agent Inference — uses Bearer or no-op auth, normalizes URL to https://{url}/api/v1
const agentClient = createAgentInferenceClient("my-agent.example.com", "optional-token");
```

**`ServiceManager`** — a registry that holds named `ServiceClient` instances:

```typescript
const manager = createServiceManager({
    digitalOcean: "do-api-key",
    serverlessInference: { token: "inference-token", baseUrl: "https://inference.do-ai.run" },
    agentInference: { url: "my-agent.example.com", token: "agent-token" },
});

const doClient = manager.getService("digitalocean");
const inferenceClient = manager.getService("serverless-inference");
```

**Internal `BearerTokenAuthenticationProvider`** — a simple Kiota `AuthenticationProvider` implementation that adds `Authorization: Bearer {token}` to requests. Used by the inference client factories. Not exported from the package (internal only).

**`ServiceConfig` interface:**

```typescript
interface ServiceConfig {
    type: ServiceType;        // DIGITALOCEAN_API | SERVERLESS_INFERENCE | AGENT_INFERENCE
    baseUrl: string;
    apiKey?: string;          // For DO API
    authToken?: string;       // For inference services
    headers?: Record<string, string>;  // Custom headers (agent inference)
}
```

### Other changes in this PR

**`DigitalOceanApiKeyAuthenticationProvider.ts`** — Now imports `package.json` and adds a `User-Agent: DigitalOcean-Dots/{version}` header to every request. This helps DigitalOcean's API team identify SDK traffic.

**`eslint.config.js`** — Adds `argsIgnorePattern: '^_'` and `varsIgnorePattern: '^_'` to the `@typescript-eslint/no-unused-vars` rule, so parameters like `_additionalAuthenticationContextProvider` (required by the interface but unused) don't trigger lint errors.

**`.js` sibling files** — Every new `.ts` file has a hand-written `.js` counterpart. This is a pattern the repo uses because `tsc` emits JS next to TS (no `outDir`), so the generated JS and hand-written JS coexist. The `.js` files are functionally identical to the `.ts` files minus type annotations.

## Part 5 — Known Design Issues in This PR

1. **`createDigitalOceanClient` name collision** — `services/index.ts` exports `createDigitalOceanClient(apiKey: string)` which collides with the Kiota-generated `createDigitalOceanClient(requestAdapter)` in `digitalOceanClient.ts`. Both are re-exported from `index.ts` via `export *`, which will cause ambiguity or errors.

2. **Streaming bypasses Kiota** — `StreamingRequestAdapter.stream()` calls `fetch()` directly instead of using the underlying adapter. This means Kiota middleware, retry policies, and auth injection don't apply to streaming requests. Headers must be pre-populated on the `RequestInformation` object.

3. **No abort support** — The streaming adapter provides no way to cancel an in-progress stream via `AbortController`.

4. **SSE parsing is simplified** — Only handles `data:` lines; ignores `event:`, `id:`, `retry:`, and multi-line `data:` per the SSE specification.

5. **Silent error swallowing** — The `emitData` method catches parse errors silently without invoking the `onError` callback.

6. **Custom headers not applied** — `createAgentInferenceClient` stores `customHeaders` in the config but never applies them to the `FetchRequestAdapter`.

7. **`assert` syntax deprecated** — `import ... assert { type: "json" }` should be `import ... with { type: "json" }` per TC39.
